# VicNail 官網整合與優化路線圖（2026-05-30）

> 目標：把「庫存管理 + 代購/團購下單」整合進官網 vicnail-studio，成為**單一**可賣、可管庫存、可上線的系統。Vic 要求：徹底完善官網、全自動完成、並規劃後續優化/升級/除錯。

---

## 一、決策

- **整合到官網**（不維持兩個分開的網站）。vic-daigou 的庫存管理併入 vicnail-studio。
- 理由：官網要能賣（隨時開團購/代購、學生直接下單付費、超商/宅配），一旦能賣就需要庫存管理；分兩套會重複。商品/訂單/庫存/金流**單一來源**最乾淨。

## 二、現況盤點（官網已經有的 vs 缺的）

**官網已具備**（Next.js 14 + Supabase + ECPay + Vercel）：
- 商店：`/shop` `/cart` `/checkout` `/checkout/success`、會員 `/account/orders`。
- 商品類型：`instock`(現貨) / `preorder`(預購) / `proxy`(代購)。
- 訂單：下單即**原子扣庫存**（防競態），ECPay 付款（信用卡/ATM/超商代碼）。
- 物流：超商 `cvs_711/fami/hilife/ok`、宅配 `home_tcat/post/sf`(順豐)、`self_pickup`(自取)。
- 團購：`group_buys` + 後台 `/admin/group-buys` + 前台 banner。
- LINE：`/api/line/webhook` 解析群組 `#下單` 自動建單。
- 後台：`/admin/{products,orders,group-buys,courses,promo-codes}`。

**缺、要補的**：
1. **庫存管理深度**：盤點、異動 ledger、低庫存警示、進貨/調整（vic-daigou 的核心）。→ 已寫 migration `009_inventory_management.sql`。
2. **後台 `/admin/inventory`** 盤點頁。
3. 訂單取消時**庫存回補** + 記異動。
4. 商品**簡稱搜尋**（autocomplete）、貨號(sku) 欄位。
5. 會員 `profiles` 加 **address + LINE 名稱**（目前只有 display_name/phone）。
6. 代購**自填商品**下單（官網目前只能選目錄商品）。
7. 7-11 **賣貨便**（C2C）若要，需額外串接（現有是 ECPay B2C 超商取貨）。

## 九、程式碼檢討發現（2026-05-30，我實際讀官網程式碼找出的問題）

> 你說「沒看到我檢討 APP、沒提出除錯資料」——這裡是具體發現，依嚴重度排序。前兩項正好是庫存整合要修的真 bug。

1. **🔴【庫存整合必修】未付款 / 付款失敗的訂單沒有回補庫存**
   - 現況：`orders/create` 下單當下就扣現貨庫存；但 `payment/ecpay/callback` 付款**失敗**(RtnCode≠1) 時只記 log，**沒取消訂單也沒把庫存加回**；棄單(建立後一直沒付)也沒清理。
   - 後果：幻影庫存流失——明明沒賣出，庫存卻一直減少，最後造成你最在意的「庫存錯亂」。
   - 修法（整合時做）：付款失敗 → 自動取消訂單 + 回補庫存 + 記 `RESTORE` 異動；加排程清理逾時未付訂單並回補。

2. **🔴【庫存整合必修】庫存變動沒有紀錄（無稽核軌跡）**
   - 現況：改 `products.stock` 沒留任何紀錄，無法回溯誰在何時為何加減。
   - 修法：已用 migration 009 的 `stock_movements`；並在「下單扣庫存 / 付款失敗回補 / 後台盤點進貨調整」各處寫入 SALE/RESTORE/STOCKTAKE/PURCHASE/ADJUST。

3. **🟡【金流強化】付款 callback 沒核對金額**
   - 現況：付款成功只驗 CheckMacValue 簽章，沒比對 `TradeAmt` 是否等於訂單金額。
   - 建議：加一行金額核對（縱深防禦），不符就標記異常不放行出貨。

4. **🟡【安全建議】後台登入用的是弱自訂雜湊**
   - 現況：`admin-auth.ts` 的 `simpleHash` 非加密雜湊、cookie 無過期、字串比對非常數時間。
   - 建議：改用 HMAC-SHA256（跟 vic-daigou 一樣）+ 設過期時間。真正防線仍是 `ADMIN_PASSWORD` 保密 + httpOnly cookie，所以不急，但值得升級。

> 註：以上是讀「金流 callback + 後台驗證」兩個最高風險檔案的結果。等資料庫能跑，我會把整站逐流程跑一遍補更多。
