## Running One AI Agent Is Easy. Running Three Is a Different World.

If you only use a single AI chatbot, things are simple. But when you run three or more AI agents simultaneously — a dispatcher assigning tasks, an executor running them, a bridge syncing state — crashes are not a question of "if" but "when."

This article tells the story of how we built **AI System Guardian** — a self-healing monitoring engine — after surviving **11 rounds of debugging and fixing 70+ production bugs**.

---

## How the Disasters Started

Our system has four core components, all managed by PM2:

1. **Orchestrator** — reads the task backlog and dispatches to workers
2. **Logic Executor** — picks up task files and runs AI-generated Python scripts
3. **Bridge** — syncs task state between two subsystems
4. **Shadow Sentinel** — the monitoring daemon itself

When everything works, it's a beautiful pipeline. When something breaks, the cascading failures are spectacular.

---

## Round 1: The Unicode Massacre (35,000 Restarts)

**Symptom:** One process had 35,000+ PM2 restarts.

**Root cause:** `subprocess.run()` on Windows defaults to the system encoding (CP-950 for Traditional Chinese). When a subprocess outputs unencodable characters, Python throws `UnicodeEncodeError`, the process crashes, PM2 restarts it, it hits the same output — infinite loop.

**Fix:** One line per subprocess call:

```python
subprocess.run(cmd, encoding="utf-8", errors="replace")
```

**What the sentinel learned:** PM2 restart anomaly detection. Any process exceeding 50 restarts gets flagged immediately.

---

## Round 2: The Bridge Infinite Loop (4,000+ Restarts)

**Symptom:** The bridge process kept restarting, maxing out CPU.

**Root cause:** Bidirectional sync created an infinite loop. Task A syncs from matrix to paperclip, triggers a change event, syncs back to matrix, triggers another event — forever.

**Fix:** Add a `source` field. If the task was last modified by the bridge itself, skip the sync.

**What the sentinel learned:** The restart anomaly check catches this pattern. Any process oscillating between "online" and "errored" faster than the cooldown period gets escalated to a human.

---

## Round 3: The Security Scan Self-Destruct Loop

**Symptom:** The security scanner also entered a 35,000+ restart loop.

**Root cause:** When the scanner found a dangerous pattern (like `eval()`), it raised an exception but never moved the file. On restart, it scanned the same file, found the same pattern, crashed again.

**Fix:** Log findings and return them instead of crashing. Move dangerous files to quarantine.

**What the sentinel learned:** The 21-pattern deep code scanner runs as a "check," not a "gate." It logs problems and notifies humans but never crashes the pipeline.

---

## Round 4: The Task Explosion

**Symptom:** Hundreds of duplicate tasks flooding the backlog. API costs spiking.

**Root cause:** Three compounding bugs — poll interval too short (60s), four parallel pipelines all generating tasks, and deduplication checking IDs that were always unique.

**Fix:**
- Poll interval from 60s to 1800s
- Date-based dedup: same type + same day = skip
- `MAX_DOING_TASKS = 6` cap

**What the sentinel learned:** Budget burn rate monitoring. Track API costs, calculate hourly burn rate, project monthly spend. Alert before overspend.

---

## Round 5: The Complete Pipeline Stall (6 Hours)

**Symptom:** All tasks stuck in "doing" status for 6+ hours.

**Root cause:** An invalid task entered the pipeline. Every worker picked it up, failed silently, left it in "doing." With `MAX_DOING_TASKS` full, nothing else could proceed.

**Fix:** Three safeguards:
1. Timeout rollback (stuck > 30 min = roll back)
2. Auto-advance (3 failures = mark as failed)
3. Retry limit (max 3 per task)

**What the sentinel learned:** Pipeline stall detection — health check #5. Runs every 10 seconds, catches stalls within 30 minutes.

## The 21 Code Scan Patterns

After the first rounds, we realized the most dangerous bugs came from AI-generated code executing without inspection. The 21-pattern scanner was born.

### Categories

| Category | Patterns | Examples |
|----------|----------|---------|
| Code Injection | 4 | eval(), exec(), \_\_import\_\_(), importlib |
| Shell Injection | 5 | os.system(), os.popen(), subprocess.Popen() |
| Native Code / Deserialization | 3 | ctypes, pickle.load(), shelve.open() |
| Filesystem / I/O | 2 | shutil.rmtree(), input() |
| Secret Access | 4 | .key files, .env files, token files, webhook JSON |
| Network Allowlist | 3 | urllib, requests, hardcoded URLs |

Every finding is logged to a security report and reported to humans, but **never interrupts the pipeline**.

## The 13 Runtime Health Checks

Each check runs every 10 seconds and returns "ok/fail" with diagnostic details:

1. **PM2 Process Health** — detects crashed processes, auto-restarts
2. **PM2 Restart Anomaly** — flags 50+ restarts
3. **Dashboard HTTP** — probes backend endpoint, restarts on failure
4. **Gateway Health** — checks port liveness
5. **Pipeline Stall** — rolls back tasks stuck > 30 minutes
6. **Orchestrator Activity** — escalates on 1-hour silence
7. **Task Quality** — pauses on 5 consecutive template fallbacks
8. **Disk Space** — warns below 500 MB
9. **Cursor Queue** — monitors queue depth
10. **Stuck Tasks** — reassigns long-running "doing" tasks
11. **Budget Burn Rate** — alerts on projected overspend
12. **Log Rotation** — auto-rotates logs > 15 MB
13. **AI Validation Health** — monitors validation pass rates

## The Cooldown System: Preventing Restart Storms

Auto-healing without rate limiting turns one crash into a thousand. Our cooldown:

- After restarting a process, **no restart for 5 minutes**
- If it crashes again within cooldown, the restart didn't fix the root cause
- Escalate to a human instead of continuing to restart

After successful recovery (process stays healthy), the cooldown timer resets so future failures can be healed immediately.

## For Those Who Want to Build Their Own

You don't need our exact tech stack. The core patterns are universal:

1. **Patrol loop** — fixed interval (10s works well), run all checks
2. **Auto-heal + cooldown** — restart failed processes but rate-limit to prevent storms
3. **Escalation chain** — auto-fix, then alert, then human task
4. **Code scanning** — AI-generated code must be scanned before execution
5. **Budget monitoring** — track burn rate for paid API calls
6. **Health summary** — write JSON every minute for dashboards

The key insight: **every check and every scan pattern exists because something actually broke in production.** Don't try to predict every failure mode upfront. Deploy the sentinel, run it, and let bugs teach you what to monitor.
