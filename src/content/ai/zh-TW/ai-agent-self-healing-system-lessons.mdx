## 跑一個 AI Agent 很簡單，跑三個以上是另一個世界

如果你只用一個 AI 聊天機器人，事情很單純。但當你同時運行三個以上的 AI Agent——調度器派工、執行器跑任務、橋接器同步狀態——崩潰就不是「會不會發生」的問題，而是「什麼時候發生」。

這篇文章講的是我們如何在經歷了 **11 輪 Debug 馬拉松、修復 70+ 個生產環境 Bug** 之後，打造出一套 **AI System Guardian（影子哨兵）**——一個能自動巡查、自動修復的監控引擎。

---

## 災難是怎麼開始的

我們的系統有四個核心元件，全部用 PM2 管理：

1. **Orchestrator（調度器）**——讀取任務積壓清單，分派給工作者
2. **Logic Executor（執行器）**——接收任務檔，執行 AI 生成的 Python 腳本
3. **Bridge（橋接器）**——在兩個子系統之間同步任務狀態
4. **Shadow Sentinel（影子哨兵）**——監控守護程式本身

一切正常時，這是一條漂亮的流水線。一旦出事，連鎖崩潰相當壯觀。

---

## 第 1 輪：Unicode 大屠殺（35,000 次重啟）

**症狀：** 一個進程重啟了 35,000+ 次。

**根因：** Windows 上的 `subprocess.run()` 預設使用系統編碼（繁體中文的 CP-950）。當子程序輸出的字元無法編碼時，Python 拋出 `UnicodeEncodeError`，進程崩潰，PM2 重啟它，又碰到同樣的輸出，再次崩潰——無限循環。

**修復：** 每個 subprocess 呼叫加一行：

```python
subprocess.run(cmd, encoding="utf-8", errors="replace")
```

**哨兵學到的：** PM2 重啟次數異常偵測。任何進程超過 50 次重啟，立刻告警。

---

## 第 2 輪：橋接器無限迴圈（4,000+ 次重啟）

**症狀：** 橋接器程式不斷重啟，CPU 爆滿。

**根因：** 雙向同步造成無限迴圈。任務 A 從矩陣同步到 Paperclip，觸發變更事件，又同步回矩陣，又觸發事件——永無止境。

**修復：** 加入 `source` 欄位。如果任務上次是被橋接器修改的，就跳過同步。

**哨兵學到的：** 重啟異常檢查能抓住這種模式。任何進程在冷卻期內反覆崩潰，就升級為人工介入。

---

## 第 3 輪：安全掃描自爆迴圈

**症狀：** 安全掃描器也進入 35,000+ 次重啟迴圈。

**根因：** 掃描到危險模式（例如 `eval()`）時直接拋出例外，但沒有把檔案移走。重啟後掃描同一個檔案，同樣的錯誤，同樣的崩潰。

**修復：** 記錄發現結果並回傳，不要崩潰。把危險檔案移到隔離區。

**哨兵學到的：** 21 模式深度掃描器作為「檢查」而非「閘門」運行。記錄問題、通知人類，但永遠不讓自己崩潰。

---

## 第 4 輪：任務大爆炸

**症狀：** 數百個重複任務湧入積壓清單。API 費用飆升。

**根因：** 三個 Bug 同時發作——輪詢間隔太短（60 秒）、四個平行管線各自產生任務、去重檢查用的是任務 ID 但每次都產生新 ID。

**修復：**
- 輪詢間隔從 60 秒改為 1800 秒
- 加入日期去重：同類型 + 同一天 = 跳過
- 加入 `MAX_DOING_TASKS = 6` 上限

**哨兵學到的：** 預算燒率監控。追蹤 API 呼叫成本，計算每小時燒率，預測月花費。超標時立刻告警。

---

## 第 5 輪：流水線全面停擺（6 小時）

**症狀：** 所有任務卡在「doing」狀態超過 6 小時，完全無進展。

**根因：** 一個配置無效的任務進入流水線。每個工作者都撿起來、處理失敗、靜默放下——但沒有改變狀態。`MAX_DOING_TASKS` 額滿，新任務全部排隊。

**修復：** 三重防護：
1. 超時回滾（30 分鐘卡住就退回）
2. 自動推進（失敗 3 次就標記失敗）
3. 重試上限（每任務最多 3 次）

**哨兵學到的：** 流水線停滯偵測是第 5 號健康檢查。每 10 秒跑一次，30 分鐘內就能抓住停滯。

---

## 21 種程式碼掃描模式

經歷前面幾輪後，我們意識到最危險的 Bug 來自 **AI 自動生成的程式碼直接執行卻沒有檢查**。21 模式掃描器就此誕生。

### 分類一覽

| 類別 | 模式數 | 範例 |
|------|--------|------|
| 程式碼注入 | 4 | eval(), exec(), \_\_import\_\_(), importlib |
| Shell 注入 | 5 | os.system(), os.popen(), subprocess.Popen() |
| 原生程式碼 / 反序列化 | 3 | ctypes, pickle.load(), shelve.open() |
| 檔案系統 / I/O | 2 | shutil.rmtree(), input() |
| 密鑰存取 | 4 | .key 檔、.env 檔、token 檔、webhook JSON |
| 網路白名單 | 3 | urllib、requests、硬編碼 URL |

每個掃描到的問題都會記錄到安全報告，通知人類審查，但**不會中斷流水線**。

## 13 項運行時健康檢查

每項檢查每 10 秒執行一次，回傳「正常 / 異常」和診斷訊息：

1. **PM2 進程健康** — 偵測崩潰進程，自動重啟
2. **PM2 重啟異常** — 超過 50 次重啟就告警
3. **儀表板 HTTP** — 探測後端端點，失敗就重啟
4. **Gateway 健康** — 檢查 port 存活
5. **流水線停滯** — 任務卡超過 30 分鐘就回滾
6. **調度器活躍度** — 1 小時無日誌就升級
7. **任務品質** — 連續 5 次模板 fallback 就暫停
8. **磁碟空間** — 低於 500 MB 就告警
9. **Cursor 佇列** — 監控佇列深度
10. **卡住任務** — 長時間 doing 就換人
11. **預算燒率** — 預測超標就告警
12. **日誌輪替** — 超過 15 MB 自動輪替
13. **AI 驗證健康** — 監控驗證通過率

## 冷卻機制：防止重啟風暴

自動修復如果沒有速率限制，會把一次崩潰變成一千次崩潰。我們的冷卻機制：

- 重啟一個進程後，**5 分鐘內不會再重啟它**
- 如果進程在冷卻期內又崩潰，代表重啟沒有解決根本問題
- 此時升級為人工介入，而不是繼續重啟

冷卻成功後（進程穩定運行），冷卻計時器重置，未來的異常能立即被處理。

## 部署後的成果

部署 System Guardian 之後：

- **偵測時間** 從數小時降到 **10 秒**
- **PM2 崩潰恢復** 從手動介入降到 **自動化（< 30 秒）**
- **流水線停滯** 從 6+ 小時降到 **30 分鐘內解決**
- AI 生成程式碼造成的安全事件：**零**（掃描器已攔截 12 個危險腳本）
- API 費用意外：**被消滅**——燒率告警在超標前就觸發
- 系統已連續運行 **5+ 天**，標準操作零人工介入

## 給想自己做的人

你不需要跟我們一樣的技術棧。核心模式是通用的：

1. **巡邏迴圈** — 固定間隔（10 秒夠好）跑所有檢查
2. **自動修復 + 冷卻** — 重啟失敗進程但限速，防止風暴
3. **升級鏈** — 自動修復 → 告警 → 人工任務
4. **程式碼掃描** — AI 生成的程式碼，執行前必掃
5. **預算監控** — 呼叫付費 API 就要追蹤燒率
6. **健康摘要** — 每分鐘寫一個 JSON，讓儀表板消費

關鍵洞察：**每一個檢查、每一個掃描模式，都是因為某個東西在生產環境真的壞掉了才存在的。** 不要試圖預判所有故障模式。先部署哨兵，跑起來，Bug 會教你該監控什麼。
